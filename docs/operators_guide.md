# AVR Assembly Revelator (v4.0)
## Advanced Static Analysis & Optimization Pipeline Operator's Guide

### 1. Executive Summary & Core Value Proposition

The **AVR Assembly Revelator (v4.0)** is an engineering-grade static analysis tool designed specifically for firmware engineers, security auditors, and performance optimizers targeting Atmel/Microchip 8-bit AVR® RISC microcontrollers (e.g., ATmega328P, ATtiny85, ATxmega).

Unlike traditional accumulator or banked memory architectures (such as the PIC® family), the 8-bit AVR architecture features a highly orthogonal, single-cycle execution engine built around a fast-access register file containing 32 general-purpose registers (r0 to r31). However, the efficiency of compiled C code generated by `avr-gcc` is highly sensitive to register allocations, stack frame operations, and interrupt service routine (ISR) prologue/epilogue setups.

Modern compilers, while highly advanced, often exhibit distinct behaviors depending on optimization levels (-O0, -O2, -Os):
* **Stack Overhead:** Stack frames are generated defensively for local variables, which requires writing to the 16-bit Stack Pointer (SPH:SPL) with global interrupts temporarily disabled.
* **Redundant Reloads:** Immediate values are frequently loaded via LDI into upper registers (r16 to r31) in close proximity to logic jumps.
* **ISR Vulnerabilities:** High-priority interrupt service routines must preserve the hardware Status Register (SREG) to prevent background thread corruption.

The AVR Assembly Revelator acts as an automated audit layer. By digesting .s (assembler output) or .lst files (generated via avr-objdump -S), it maps Special Function Registers (SFRs), validates frame-by-frame stack operations, assesses interrupt safety, and highlights opportunities to reclaim processing cycles and Flash space.

### 2. Architecture & Data Pipeline Breakdown

Version 4.0 utilizes a robust, state-isolated multi-stage ingestion pipeline designed specifically for the characteristics of AVR RISC instructions and GNU assembler formatting.

#### Stage 1: Disassembly & Assembler Parser
The parser supports two main formats:
1. **Raw Assembly Source (.s / .asm):** Ingests GCC directive flags (.global, .section, .quad), parsing mnemonic tokens and local labels (such as .L2).
2. **Objdump Disassembly Listings (.lst / .txt):** Processes output generated via `avr-objdump -S -d` or `avr-objdump -h`. It isolates line numbers, absolute 16-bit hex addresses, and raw hexadecimal opcode bytes before parsing the clean assembly instruction.

#### Stage 2: AVR Application Binary Interface (ABI) State Engine
The tool tracks register operations and dynamically maintains live state structures. For AVR, register-tracking rules are highly specific:
* **The LDI Restriction:** Immediate loads (LDI) are restricted to registers r16 to r31.
* **AVR ABI Call-Clobbered Register Rules:** The tool assumes any standard function execution via CALL or RCALL immediately invalidates (clobbers) the following registers:
  r0, r1, r18, r19, r20, r21, r22, r23, r24, r25, r26, r27, r30, r31
* **16-bit Register Pairs:** The movw instruction copy-actions are tracked as double 8-bit modifications (e.g., copying r22 -> r24 automatically flags both r24 and r25 as modified).

#### Stage 3: Special Function Register (SFR) Mapping
The engine maps raw numeric ports and I/O addresses used in I/O instructions (in, out, sbi, cbi) back to their symbolic representations.
* **Direct I/O Address Translation:** Port address 0x3F is mapped back to SREG, while 0x3D/0x3E resolve to stack registers SPL/SPH.
* **Data Space Memory Translation:** Addresses are normalized to highlight direct-space manipulation (such as lds r24, 0x00C0 mapping to UDR0).

### 3. Comprehensive Metric Diagnostic Dictionary

The diagnostic suite evaluates AVR binaries against seven critical safety and optimization heuristics.

#### 1. REPEATED_LDI (Severity: WARN)
* **Trigger Condition:** An LDI instruction loads an immediate value into a high register (r16 to r31) that matches the value already loaded by a preceding LDI or CLR/SER command, with no intervening operations modifying that target register.
* **Remediation:** If executing within tight timing loops, reorganize the assignments in your C source to allow the compiler to reuse register variables.

#### 2. LDI_ZERO_OPTIMIZATION (Severity: INFO)
* **Trigger Condition:** An LDI instruction is used to clear a register to 0 (LDI Rd, 0).
* **Remediation:** Replace with CLR Rd (compiled as EOR Rd, Rd) to save program space and allow register utilization across the entire file descriptor set.

#### 3. PUSH_POP_MISMATCH (Severity: STRONG)
* **Trigger Condition:** A function boundary is identified where the number of stack PUSH instructions does not equal the number of stack POP instructions.
* **Remediation:** **CRITICAL PRIORITY.** An unbalanced stack corrupts function return calls, crashing the CPU immediately. Re-audit all conditional branch paths inside the inline assembly or assembly source.

#### 4. ISR_MISSING_SREG_SAVE (Severity: STRONG)
* **Trigger Condition:** A function ending with RETI (Interrupt Return) contains no instructions to save the Status Register (SREG, I/O Address 0x3F).
* **Remediation:** **CRITICAL PRIORITY.** Saving SREG immediately inside your ISR prologue prevents arithmetic operations from corrupting the flag state of the interrupted background thread.

### 4. Command-Line Interface (CLI) Specification

Launch the AVR analyzer from your terminal shell:

```bash
python avr_asm_revelator_v4_0.py <input_file_path> [options]
```

#### CLI Configuration Flags
* `input_file_path` (Path, Required): Target file path (.lst, .s, .asm, .txt).
* `--report <path>`: Overrides default Markdown report output path.
* `--json <path>`: Generates machine-readable output.
* `--annotate`: Enables inline listing comment generation.
* `--min-severity {INFO, WARN, STRONG}`: Minimum severity threshold.
